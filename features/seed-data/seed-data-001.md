# Task: seed-data-001

## Objective

Implement comprehensive seed data generation in `priv/repo/seeds.exs` to populate the development database with realistic test data for QA purposes. The seed script must be idempotent and run automatically during devcontainer setup.

## Context

- The app name is `Acai` with the repo module `Acai.Repo`
- The seeds file (`priv/repo/seeds.exs`) is currently a stub with only placeholder comments
- The devcontainer `postcreate.sh` already runs `mix ecto.setup`, which includes `mix run priv/repo/seeds.exs` — so no changes needed there
- The `Acai.Accounts` context provides `register_user/1` for user creation (takes `%{email: ...}`)
- User confirmation is done via `User.confirm_changeset/1` which sets `confirmed_at`
- Password is set via `User.password_changeset/2` (takes user struct and `%{password: ...}`)
- The `Acai.Teams` context provides `create_team/2` (takes `current_scope` and `%{name: ...}`)
- Roles are managed via `UserTeamRole` with titles: `"owner"`, `"developer"`, `"readonly"`
- `Acai.Accounts.Scope` provides `Scope.for_user(user)` to build scope structs needed by context functions
- All UUIDs are UUIDv7 (`Acai.UUIDv7`), auto-generated by schemas
- The test fixtures in `test/support/fixtures/data_model_fixtures.ex` demonstrate how to create all entity types at the Repo level — use similar patterns in the seed file

## Acceptance Criteria

### SEED_DATA.USERS — User Accounts

- [ ] `SEED_DATA.USERS.1` — Pregenerate the following user accounts: `owner@testing.team`, `developer@testing.team`, `readonly@testing.team`
- [ ] `SEED_DATA.USERS.2` — All pregenerated users must have the password `password123456`
- [ ] `SEED_DATA.USERS.3` — All pregenerated users must be marked as confirmed

### SEED_DATA.TEAMS — Teams and Membership

- [ ] `SEED_DATA.TEAMS.1` — Generate two teams: `testing-team` and `empty-team`
- [ ] `SEED_DATA.TEAMS.2` — Assign the pregenerated users to `testing-team` with their respective roles: `owner@testing.team` → `owner`, `developer@testing.team` → `developer`, `readonly@testing.team` → `readonly`
- [ ] `SEED_DATA.TEAMS.3` — Assign the pregenerated users to `empty-team` with their respective roles: `owner@testing.team` → `owner`, `developer@testing.team` → `developer`, `readonly@testing.team` → `readonly`

### SEED_DATA.MOCK_DATA — Mock Domain Data

- [ ] `SEED_DATA.MOCK_DATA.1` — Generate at least 3 Specs for `testing-team` with varying states: one with full requirements and multiple implementations, one with pending requirements and no implementations, one with a mix of implemented/partially implemented/deprecated requirements
- [ ] `SEED_DATA.MOCK_DATA.2` — Requirements for each spec must cover: both `COMPONENT` and `CONSTRAINT` group types, nested requirements (sub-requirements via `parent_local_id`), requirements with notes and `replaced_by` ACIDs
- [ ] `SEED_DATA.MOCK_DATA.3` — Generate Implementations for the specs: at least one implementation with a linked `TrackedBranch`, varying statuses for requirements within implementations
- [ ] `SEED_DATA.MOCK_DATA.4` — Generate a history of ActivityEvents for `testing-team`: events for spec creation, requirement updates, and implementation progress; events attributed to different team members (owner, developer)

### SEED_DATA.ENVIRONMENT — Constraints

- [ ] `SEED_DATA.ENVIRONMENT.1` — Seed data generation must run automatically during devcontainer build or when `mix run priv/repo/seeds.exs` is executed
- [ ] `SEED_DATA.ENVIRONMENT.2` — Seeding must be idempotent; running it multiple times should not create duplicate users or teams

## Action Items

- [ ] Implement the full seed script in `priv/repo/seeds.exs`
  - Use `Repo.get_by` lookups (e.g., by email for users, by name for teams) to achieve idempotency — only create records that don't already exist
  - Register users via `Acai.Accounts.register_user/1`, then set passwords via `User.password_changeset/2` and confirm via `User.confirm_changeset/1` (apply with `Repo.update!`)
  - Create teams using the owner user's scope via `Acai.Teams.create_team/2` (which auto-assigns the creating user as owner)
  - Add remaining user-team roles by inserting `UserTeamRole` records directly
  - Create specs, requirements, implementations, tracked branches, requirement statuses, and activity events using the context APIs listed above
  - For activity events, note that `actor_token_id` is optional — you can create an access token for the team via `Acai.Teams.generate_token/4` to use as actor, or leave it nil
- [ ] Verify idempotency: running `mix run priv/repo/seeds.exs` twice should not raise errors or create duplicates
- [ ] Run `mix precommit` to ensure compilation and tests pass

### Findings

#### ❌ ACID comments contain extra description text (violates implement-spec rules)

The implement-spec rules are explicit: *"You must only write the ACID on its own. NEVER duplicate spec requirement text in comments."*

Many ACID comments in `priv/repo/seeds.exs` append human-readable summaries after the ACID, coupling the code to the spec text and creating the exact maintenance problem the rule exists to prevent. All of the following must be trimmed to the bare ACID(s) only:

```
# line 174 — remove " — Pregenerate the three test users"
# line 184 — remove " — Pregenerate two teams"
# line 193 — remove " — Assign users to testing-team" + the prose comment on line 194-195
# line 202 — remove " — Assign users to empty-team"
# line 212 — remove " (Spec A) — Full requirements + multiple implementations"
# line 213 — remove " — COMPONENT + CONSTRAINT groups, nested reqs, notes, replaced_by"
# line 228 — remove the "# COMPONENT group requirements" free-text comment
# line 288 — remove " — SEED_DATA.MOCK_DATA.2" suffix / rephrase as bare ACID
# line 311 — remove "# A deprecated requirement with replaced_by —" prefix; keep only the ACID

# line 336 — remove " — linked TrackedBranch"
# line 343 — remove "# Varying requirement statuses —" prefix
# line 369 — remove "# Second implementation (feature branch) —" prefix
# line 396 — remove the IO.puts line (not an ACID comment, but contextual prose is fine; this one is fine to keep as IO.puts are progress logs, not comments)

# line 635 — remove " — Activity events for testing-team"
# line 637-638 — remove the free-text comment block (prose, not ACID)
```

The corrected form for a section header is simply:
```elixir

spec_a = seed_spec.(...)
```

And for inline annotations:
```elixir
# SEED_DATA.MOCK_DATA.3
_branch_a_prod = seed_branch.(...)
```
